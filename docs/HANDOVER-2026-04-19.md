## 1. What Survai actually is

Survai is an **AI-native, voice-first survey builder**. The differentiator
vs Typeform / Google Forms is the voice respondent flow: published surveys
can be answered by speaking into a browser (ElevenLabs Conversational AI),
with post-call transcripts mapped back to the typed survey schema.

The creator-facing editor uses Anthropic Claude (Opus 4.7) to turn a plain-
English prompt into a real, typed survey schema that's rendered in a 3-panel
editor (palette + canvas + chat/properties).

**Do not trust the README on this point.** [README.md:8](../README.md#L8)
still claims OpenAI GPT-4o; the codebase moved to Anthropic Claude during
the backend hardening milestone. OpenAI is now only used for Whisper
(voice input transcription), DALL-E (background images), and as a TTS
fallback. Anthropic is the primary reasoning engine.

### Core surfaces

| Route                         | Auth      | What it is                                                                 |
| ----------------------------- | --------- | -------------------------------------------------------------------------- |
| `/`                           | anon      | Marketing landing page.                                                    |
| `/test`, `/test/edit`         | anon      | Try-before-signup editor (localStorage-only, no DB writes).               |
| `/admin`                      | anon      | **Duplicate** of `/test/edit`. See finding #6 — should be deleted.         |
| `/dashboard`                  | creator   | Survey list; gated by `src/proxy.ts`.                                      |
| `/survey/[id]/edit`           | creator   | The real editor (Supabase-backed, auto-saves).                             |
| `/survey/[id]/dashboard`      | creator   | Results + A2UI-rendered charts.                                            |
| `/survey/[id]/responses`      | creator   | Individual response viewer.                                                |
| `/s/[id]`, `/s/[id]?t=<tok>`  | anon      | Respondent surface — voice-first if an ElevenLabs agent is provisioned.    |

### Core backend pipelines

1. **AI generation**: creator prompt → `POST /api/ai/chat` → Claude (structured output via zod discriminated union) → one of four intents: `generate`, `propose`, `command`, `clarify` → either a full survey blueprint (hydrated via `src/lib/templates/hydrate.ts`) or `UiCommand[]` applied to the Zustand store.
2. **Voice respondent**: creator publishes → `POST /api/elevenlabs/agent` builds a voice-configured agent from the 17-type survey schema and stores `agent_id` on the survey row → respondent loads `/s/[id]` → `GET /api/elevenlabs/signed-url` → browser opens WebRTC session via `@elevenlabs/react` → ElevenLabs fires `POST /api/webhooks/elevenlabs` → HMAC-verified, idempotency-guarded, transcript-mapped response row inserted.
3. **Security perimeter**: `src/proxy.ts` refreshes Supabase auth cookies on every request, rate-limits all `/api/*` except `/api/health` and webhooks (in-memory per-IP sliding window), redirects anon traffic to `/login?next=…` when it touches a protected prefix.

## 3. Findings from the 2026-04-19 audit

Ordered by impact. Severity tiers: **CRITICAL** = fix before next deploy. **HIGH** = fix this week. **MEDIUM** = quality debt, fix before scaling users.

### #1 — Authenticated editor hits the anonymous demo chat route — CRITICAL ✅ fixed (`5f83488`)

[src/components/survey/editor/survey-editor.tsx:36](../src/components/survey/editor/survey-editor.tsx#L36) renders `<RightPanel />` without passing `aiEndpoint` or `aiStreamEndpoint`. [src/components/survey/chat/chat-panel.tsx:35](../src/components/survey/chat/chat-panel.tsx#L35) defaults to `/api/ai/chat/test` — the anon demo route. Consequences:

- The authenticated route [src/app/api/ai/chat/route.ts](../src/app/api/ai/chat/route.ts) is effectively **dead code** from the UI.
- `ai_traces` writes (owner-gated) and `settings.aiContext.{model, temperature, systemPromptOverride}` overrides are never exercised.
- The live DB confirms this: only 2 rows in `ai_traces` despite extensive editor testing.

**Fix**: one line. Pass `aiEndpoint="/api/ai/chat"` (and `aiStreamEndpoint="/api/ai/chat/stream"` if you want streaming in the auth flow) to `<RightPanel />`.

### #2 — `commands[].action` vs `type` schema-contract mismatch — CRITICAL-for-UX ✅ fixed (`1d2cc01`)

The AI is correctly identifying user intent for "small edit" requests, but the model emits `{type: 'delete_element', …}` while [src/lib/ai/schema.ts:111-132](../src/lib/ai/schema.ts#L111-L132) only accepts `{action: …, …}`. Zod rejects → 502 → user sees "AI response did not match schema" with no retry path.

Root cause: [src/lib/ai/prompts.ts:96-102](../src/lib/ai/prompts.ts#L96-L102) names the action *values* but never pins the field name. Model guessed `type` (a common JSON convention); coin flip lost.

Why it's masked today: most real use has been `generate` / `propose` intents. Every `command` turn ("make it shorter", "remove Q3", "move X above Y") hits this. The `command-executor` in [src/lib/ai/command-executor.ts](../src/lib/ai/command-executor.ts) is fully wired and correct — it simply never gets called because zod trips first.

**Fix**: four changes, ~30 LOC total, shipped as one commit (see §5 for the exact plan).

### #3 — `base_schema.sql` still contains the insecure guests policy — CRITICAL for fresh installs ✅ fixed (`a849841`)

[supabase/migrations/20260417000000_base_schema.sql:139-141](../supabase/migrations/20260417000000_base_schema.sql#L139-L141) keeps:

```sql
create policy "Public read by token" on public.guests
  for select using (true);
```

The `using (true)` means any holder of the anon key can dump every guest row (tokens, names, emails, phones). Migration `20260419000100_rls_guests_token.sql` drops it, but [docs/SETUP.md:37-41](./SETUP.md#L37-L41) tells new installers to paste only the base-schema file.

**Live DB is fine** — it received the fix separately. But anyone cloning the repo today and following SETUP is one `curl` away from leaking their guest table.

**Fix**: either inline the drop into the base-schema file and re-ship as a single canonical migration, or update SETUP to say "run every file in `supabase/migrations/` in lexical order".

| Repo (4 files)                                   | Live DB (7 applied migrations)                                  |
| ------------------------------------------------ | --------------------------------------------------------------- |
| `20260417000000_base_schema.sql`                 | `20260418031828_base_schema`                                    |

### #6 — `/test` and `/admin` are duplicate unauthenticated editors — HIGH ✅ fixed (`5f83488`, folded into #1)

[src/app/admin/page.tsx](../src/app/admin/page.tsx) is a near-identical copy of `/test/edit`, listed as PUBLIC in [src/proxy.ts:34](../src/proxy.ts#L34) with the comment "in-product try-before-signup surface". Two unauth surfaces doing the same thing, one of them named `/admin` — confusing now, dangerous later when someone builds a real admin console and assumes the name is free.

**Fix**: delete `src/app/admin/page.tsx` and remove `/admin` from `PUBLIC_PREFIXES` in [src/proxy.ts:34](../src/proxy.ts#L34).

### #7 — `/test` has no autosave + rate limiter is single-process in-memory — HIGH

Two related weaknesses on the anon public surface:
- `/test/edit` users lose their draft on refresh. This is the "try before signup" funnel; dropping work mid-build costs conversions.
- [src/lib/api/rate-limit.ts:20-24](../src/lib/api/rate-limit.ts#L20-L24) is an in-process `Map`. Cloud Run with `min-instances=0` resets counters on cold start. A patient attacker can pause ~15 minutes and refill the quota. `SECURITY.md §5` acknowledges this as a v1 tradeoff, but `/api/ai/chat/test` is your most expensive public surface (Claude Opus calls) with a 200-req/hr/IP cap.

**Fix**: (a) localStorage autosave in `/test/edit` with a "Save to account" CTA that lifts to the authenticated flow. (b) Move rate limiter to Redis (or Supabase Postgres with a small `rate_limit_hits` table) before you raise `min-instances` above 0 *or* open the firehose to non-beta users.

### #8 — Env fail-fast misses `ANTHROPIC_API_KEY` and the ElevenLabs secrets — MEDIUM

[src/lib/env.ts](../src/lib/env.ts) only requires the Supabase pair + `OPENAI_API_KEY` at prod boot. Since Anthropic is the *primary* AI, a Cloud Run revision deployed without `ANTHROPIC_API_KEY` will start healthy, pass `/api/health`, and then 500 silently on every editor turn.

Same class of bug for `ELEVENLABS_API_KEY` (voice agent provisioning fails) and `ELEVENLABS_WEBHOOK_SECRET` (every incoming webhook 500s with "invalid signature").

**Fix**: extend `REQUIRED_SERVER_VARS` to include all three. For `SUPABASE_SERVICE_ROLE_KEY` — which the app is designed to tolerate the absence of — emit a warn-at-boot log line so missing-secret deploys are at least visible in Cloud Logging.

Also update [src/app/api/ai/health/route.ts](../src/app/api/ai/health/route.ts) to report `{openai, anthropic, elevenlabs, supabaseServiceRole}` instead of just `openaiConfigured`.

### #9 — API error envelope is inconsistent — MEDIUM

[docs/BACKEND-API.md:7](./BACKEND-API.md#L7) promises a typed envelope `{error, code, details?}`. `/api/surveys/*` honors it. `/api/ai/*`, `/api/webhooks/elevenlabs`, and `/api/elevenlabs/*` mostly return ad-hoc `NextResponse.json({error: '...'})`. The frontend can't rely on the `code` field being present.

**Fix**: migrate the ad-hoc routes to [src/lib/api/errors.ts](../src/lib/api/errors.ts) helpers. No behavior change; just consistency.

### #10 — `chat_messages` table is a ghost — MEDIUM

Table exists, RLS'd, indexed. Live DB shows 0 rows, 0 writers in `src/`. The "conversation history per survey" feature is a leftover from a parallel workstream that never landed.

**Fix**: either (a) delete the table + migration, or (b) actually persist `chatMessages` from the Zustand store on turn completion. Today it's maintenance overhead with zero behavior.

### #12 — `POST /api/elevenlabs/agent` is intentionally un-gated; docs claim 401 — MEDIUM (doc drift)

Surfaced while walking §8 locally on 2026-04-19 (follow-up session).

- [§8 pre-deploy checklist](#8-pre-deploy-verification-checklist) claims
  `curl -X POST /api/elevenlabs/agent -d '{}' → 401`.
- [SECURITY.md §1 threat matrix](../SECURITY.md) row for
  `/api/elevenlabs/agent, /batch, /call, /tts` claims "creator;
  `requireAuth()` — 401 for anon".
- **Reality**: only `PATCH` and `DELETE` on `/api/elevenlabs/agent` are
  auth-gated ([src/app/api/elevenlabs/agent/route.ts:64](../src/app/api/elevenlabs/agent/route.ts#L64)
  and `:101`). `POST` is deliberately ungated so the /test share-link
  flow can mint a voice agent without requiring signup — see commit
  `fa9e98c` ("fix(voice): unblock /test share-link voice by un-gating
  agent creation"). Cost exposure is bounded by the proxy rate limit
  (default 100/hr/IP on that path) plus ElevenLabs' own quotas.

Not a vulnerability — a conscious tradeoff. But the pre-deploy check
will never turn green as written, and a new engineer reading
SECURITY.md will draw the wrong conclusion about the threat model.

**Fix**: update §8 to expect `400 "Survey must have at least one
element"` (i.e., prove the route shortcircuits on invalid payload, not
on auth), and update the SECURITY.md row to split POST (anon,
rate-limited) from PATCH/DELETE (creator-only). If the ungate is only
meant to last until a proper "demo mode" flag exists, file a
follow-up to gate it on `demoMode=true` once that flag lands.

## 5. Ready-to-implement plan: fix the command-schema bug (finding #2)

Ship all four changes as one commit. ~30 LOC total.

### Change 1 — Pin the JSON shape in the prompt

[src/lib/ai/prompts.ts:96-102](../src/lib/ai/prompts.ts#L96-L102). Replace the current "command" intent description with:

```
**"command"** — Small edits. Return commands[] where each command is
{ action: <name>, ...params }. Allowed actions:
  - action:'move_element' + elementId + toIndex
  - action:'update_element' + elementId + updates
  - action:'delete_element' + elementId
  - action:'duplicate_element' + elementId
  - action:'update_settings' + settings
  - action:'select_element' + elementId
  - action:'publish_survey' + respondentCount?
```

While you're in this file, grep for the other three intents (`generate`, `propose`, `clarify`) and ensure each field name the AI must emit is stated explicitly, not implied.

### Change 2 — Defensive alias in the zod schema

[src/lib/ai/schema.ts:111-132](../src/lib/ai/schema.ts#L111-L132). Wrap `uiCommandSchema` (the inner schema — not `aiResponseSchema`) in a `z.preprocess` that normalizes both directions:

```ts
export const uiCommandSchema = z.preprocess((val) => {
  if (val && typeof val === 'object') {
    const obj = val as Record<string, unknown>;
    if ('type' in obj && !('action' in obj)) {
      const { type, ...rest } = obj;
      return { action: type, ...rest };
    }
  }
  return val;
}, z.object({ /* existing discriminated union */ }));
```

Accepting `type → action` handles the documented failure mode. Symmetric handling (`action → type` if the model ever drifts back) is optional; I'd skip it to keep the schema the canonical shape.

### Change 3 — Better user-facing error UX

[src/hooks/use-ai-chat.ts](../src/hooks/use-ai-chat.ts) + the chat error message component. Two-part fix:

**(a) Client side**: when the API returns a schema-parse error, render an actionable message with a "Retry" button that resends the last user turn. Replace the current "Something went wrong: AI response did not match schema" with something like:
> The agent's reply didn't land. Want me to retry, or can you say more specifically what to change?

**(b) Server side**: in [src/app/api/ai/chat/route.ts](../src/app/api/ai/chat/route.ts), on `safeParse` failure, do one silent retry with a nudge appended to the next turn ("Your previous reply didn't match the required JSON shape; emit `commands[]` with `action` fields per the system prompt"). Only surface the error if the retry also fails. 400-500ms latency on a rare failure path is worth not dead-ending the user.

### Change 4 — Regression test

[src/lib/ai/__tests__/schema.test.ts](../src/lib/ai/__tests__/schema.test.ts). Add two cases:
1. Parsing a command with `type` (alias) succeeds and normalizes to `action`.
2. Parsing a command with an unknown `action` value (e.g. `"destroy_element"`) still rejects cleanly — the alias isn't a bypass for typo'd actions.

## 8. Pre-deploy verification checklist

Run these before promoting any new revision to production. Most are already in [SECURITY.md §6](../SECURITY.md#L200) — reproduced here as a single pass:

- [ ] `curl -X POST https://sur-ways.com/api/ai/responses -d '{}'` → 401
- [ ] `curl -X POST https://sur-ways.com/api/elevenlabs/agent -d '{}'` → 400 `"Survey must have at least one element"` (POST is intentionally ungated — see finding #12; the check proves the route is reachable without becoming an abuse vector)
- [ ] `curl https://sur-ways.com/api/webhooks/elevenlabs -X POST -d '{}' -H 'elevenlabs-signature: t=0,v0=0'` → 401
- [ ] Hammer `/api/ai/chat/test` 201 times from a single IP → 429 on request 201
- [ ] `curl $SUPABASE_URL/rest/v1/guests -H "apikey: $ANON_KEY"` → `[]` (confirms finding #3 is not leaking into the live DB)
